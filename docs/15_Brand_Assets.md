# HRNEIROWAY 2026 — Brand Assets

## 1. Purpose

Этот документ фиксирует правила будущей библиотеки бренд-материалов HRNEIROWAY 2026.

Он нужен, чтобы все визуальные активы проекта хранились системно, назывались понятно и поддерживали единый художественный уровень сайта.

Документ фиксирует:

* какие бренд-материалы нужны проекту;
* где они хранятся;
* как их называть;
* какие требования к качеству;
* что можно использовать на сайте;
* что нельзя использовать;
* как работать с референсами;
* как хранить промпты;
* как архивировать старые версии.

Главный принцип:

бренд-материалы должны усиливать HR Editorial Workspace, а не превращать проект в набор случайных красивых файлов.

## 2. Brand Materials Folder Structure

Будущие бренд-материалы должны храниться в папке:

```text
brand-materials/
├── logos/
├── icons/
├── textures/
├── mockups/
├── prompts/
├── photos/
│   ├── marina/
│   ├── workspace/
│   └── documents/
└── references/
    ├── websites/
    ├── layouts/
    ├── editorial/
    ├── typography/
    └── animations/
```

### `logos/`

Папка для всех версий логотипа HRNEIROWAY:

* основной логотип;
* горизонтальная версия;
* компактная версия;
* favicon;
* SVG;
* PNG;
* светлая и тёмная версии.

### `icons/`

Папка для фирменных иконок и маленьких графических элементов:

* документы;
* папки;
* чек-листы;
* презентации;
* стрелки;
* закладки;
* AI-помощник;
* диагностика.

### `textures/`

Папка для текстур, которые поддерживают материальность сайта:

* бумага;
* хлопковая бумага;
* картон;
* лен;
* дерево;
* мягкий шум;
* матовые поверхности.

### `mockups/`

Папка для макетов и визуализаций материалов:

* welcome book mockups;
* инструкции;
* презентации;
* PDF-гайды;
* экраны AI-инструментов;
* материалы “до/после”.

### `prompts/`

Папка для промптов, которые используются для генерации или подготовки визуальных материалов.

Здесь должны храниться не только сами промпты, но и пояснения: для чего промпт, какие цвета использовать, что запрещено и где результат будет применяться.

### `photos/`

Папка для всех фотографий проекта.

Внутри:

* `marina/` — фото Марины и рабочего процесса;
* `workspace/` — рабочий стол и атмосфера;
* `documents/` — реальные или суперреалистичные материалы.

### `references/`

Папка для референсов.

Важно: референсы нужны не для копирования, а для фиксации уровня качества, настроения, композиции, поведения и деталей.

## 3. Logo Assets

Будущая система логотипов должна включать:

* основной логотип;
* горизонтальную версию;
* компактную версию;
* favicon;
* SVG;
* PNG на прозрачном фоне;
* светлую версию;
* тёмную версию.

Правила логотипа:

* чистый;
* editorial;
* спокойный;
* без лишних эффектов;
* без 3D;
* без градиентов;
* без глянца;
* без неона;
* хорошо работает на сливочном фоне #FFFCEF;
* хорошо работает на navy фоне #13273F;
* читается в маленьком размере;
* не конфликтует с документами, фото и карточками.

Логотип должен ощущаться как часть премиального editorial workspace, а не как знак IT-стартапа, HR-портала или digital-агентства.

## 4. Signature Elements

### Бордовая закладка

Где используется:

* active menu;
* карточки услуг;
* CTA;
* кейсы;
* footer;
* важные акценты.

Какую эмоцию создаёт:

* собранность;
* фокус;
* ощущение важной страницы в документе;
* фирменную узнаваемость.

Чего нельзя делать:

* использовать слишком часто;
* делать слишком яркой;
* превращать в декоративный шум;
* использовать как случайный элемент без смысла.

### Тонкая editorial-линия

Где используется:

* между секциями;
* в process steps;
* рядом с подписями;
* для направления взгляда;
* в карточках и схемах.

Какую эмоцию создаёт:

* порядок;
* связь;
* редакционную точность.

Чего нельзя делать:

* превращать линии в сложные схемы без смысла;
* перегружать экран;
* использовать линии, которые не ведут взгляд.

### Живая стрелка

Где используется:

* hero;
* CTA;
* переходы между секциями;
* кнопки;
* полезные материалы.

Какую эмоцию создаёт:

* мягкое движение;
* приглашение продолжить;
* ощущение живого интерфейса.

Чего нельзя делать:

* заставлять стрелку дёргаться;
* делать движение резким;
* использовать слишком много стрелок на одном экране.

### Оливковая веточка

Где используется:

* hero;
* about;
* полезные материалы;
* кейсы;
* маленькие editorial-акценты.

Какую эмоцию создаёт:

* живость;
* мягкость;
* рост;
* человеческое тепло.

Чего нельзя делать:

* уводить визуал в wellness;
* использовать веточку слишком крупно;
* делать её главным декоративным элементом сайта.

### Paper label

Где используется:

* карточки;
* категории;
* полезные материалы;
* пометки;
* навигационные подписи.

Какую эмоцию создаёт:

* ощущение рабочего архива;
* порядок;
* ручную систематизацию.

Чего нельзя делать:

* делать ярлыки слишком декоративными;
* перегружать ими карточки;
* использовать нечитаемые подписи.

### Мягкая матовая плашка

Где используется:

* CTA;
* формы;
* заметки;
* выделение важных блоков;
* secondary content.

Какую эмоцию создаёт:

* тактильность;
* спокойствие;
* ощущение качественной поверхности.

Чего нельзя делать:

* превращать плашку в glassmorphism;
* использовать сильный blur;
* делать поверхность глянцевой.

### Карточка-документ

Где используется:

* услуги;
* полезные материалы;
* популярные решения;
* кейсы;
* блоки “что получится”.

Какую эмоцию создаёт:

* ощущение реального материала;
* понятность;
* рабочую применимость.

Чего нельзя делать:

* делать карточки generic;
* использовать одинаковую плитку без характера;
* перегружать hover-эффектами.

### Папка клиента

Где используется:

* кейсы;
* до/после;
* proof-блоки.

Какую эмоцию создаёт:

* доверие;
* доказательность;
* ощущение настоящего проекта.

Чего нельзя делать:

* показывать папку как пустой декоративный элемент;
* использовать без контекста задачи и результата.

### Библиотечная карточка

Где используется:

* полезные материалы;
* блог;
* чек-листы;
* промпты;
* шаблоны.

Какую эмоцию создаёт:

* желание сохранить;
* ощущение системы знаний;
* пользу до заявки.

Чего нельзя делать:

* делать карточку похожей на обычный blog card;
* не указывать пользу материала;
* использовать без понятного CTA.

## 5. Icon System

Иконки HRNEIROWAY должны быть:

* тонкие;
* спокойные;
* editorial;
* не generic;
* без мультяшности;
* без 3D;
* без неона;
* без чрезмерной технологичности.

Иконки должны поддерживать материалы и процессы:

* документ;
* папка;
* чек-лист;
* презентация;
* AI-помощник;
* коммуникация;
* диагностика;
* сайт;
* стрелка;
* закладка.

Правила:

* иконка должна быть понятной без подписи, если используется как UI-элемент;
* декоративные иконки должны быть вторичными;
* не использовать случайные наборы из разных библиотек;
* толщина линий должна быть единой;
* скругления и углы должны соответствовать общему характеру сайта;
* иконки не должны заменять реальные визуальные доказательства.

Иконка может поддержать смысл, но не должна становиться главным визуалом там, где нужен документ, фото или материал.

## 6. Texture Library

Нужны текстуры:

* бумага;
* хлопковая бумага;
* картон;
* лен;
* дерево;
* матовая поверхность;
* лёгкий шум;
* тёплая ткань.

Правила:

* текстуры должны быть едва заметными;
* не делать сайт грязным;
* не использовать грубую бумажную эстетику;
* текстура не должна мешать читаемости;
* текстура не должна спорить с фотографиями;
* текстура должна поддерживать ощущение тактильности;
* не использовать текстуру просто потому, что экран “пустой”.

Текстуры должны работать как тихий слой качества. Пользователь может не заметить их сознательно, но должен почувствовать глубину и материальность интерфейса.

## 8. References System

Референсы нужно хранить не как случайную папку скриншотов, а как рабочую библиотеку решений.

Важно:

скриншоты сами по себе недостаточны, потому что они не передают анимацию, hover и ощущения.

Поэтому рядом с референсами нужно вести текстовые заметки.

Рекомендованный формат заметки:

```md
## REF-001 — название сайта

Ссылка:
...

Категория:
hero / footer / cards / animation / typography / layout / form

Что понравилось:
...

Что НЕ брать:
...

Что берем для HRNEIROWAY:
...

Рейтинг:
★★★★★ / ★★★★☆ / ★★★☆☆
```

* общее впечатление;

* качество интерфейса;

## 9. Prompt Library

Папка:

`brand-materials/prompts`

Туда складывать:

* промпты для hero-фото;
* промпты для рабочего стола;
* промпты для документов;
* промпты для иконок;
* промпты для паттернов;
* промпты для генерации портретов;
* промпты для mockups.

Каждый промпт хранить с пояснением:

* для чего он;
* какие цвета использовать;
* какой формат кадра;
* что запрещено;
* где результат будет использоваться.

Рекомендуемая структура описания промпта:

```md
## Prompt — название

Назначение:
...

Где используется:
...

Формат:
hero / card / document / portrait / mockup / icon / texture

Цвета:
...

Обязательные элементы:
...

Запрещено:
...

Prompt:
...

Notes:
...
```

Промпты должны поддерживать HR Editorial Workspace. Нельзя хранить промпты, которые генерируют роботов, AI-сферы, неон, generic office или дешёвый AI-art.

## 10. File Naming Rules

Правила названий файлов:

* только латиница;
* lowercase;
* слова через дефис;
* без пробелов;
* без кириллицы;
* без `final-final`;
* без `image1`;
* понятные названия;
* при необходимости использовать номер версии;
* не смешивать языки в одном названии.

Примеры:

* `hrneiroway-logo-primary.svg`
* `hero-workspace-welcome-book.jpg`
* `burgundy-bookmark.svg`
* `olive-branch-line.svg`
* `welcome-book-fragment-01.jpg`
* `homepage-hero-workspace-v01.jpg`
* `case-folder-before-after-01.jpg`

Плохие примеры:

* `image1.jpg`
* `final-final.png`
* `лого новое.svg`
* `photo copy 2.jpg`
* `ai-generated-cool-thing.png`

Название файла должно объяснять, что это за материал и где его можно использовать.

## 11. Quality Checklist

Перед добавлением любого материала в проект проверять:

* соответствует ли HR Editorial Workspace;
* выглядит ли премиально;
* не похоже ли на сток;
* есть ли смысл;
* не перегружает ли сайт;
* подходит ли к палитре;
* можно ли использовать на mobile;
* не нарушает ли конфиденциальность;
* оптимизирован ли файл;
* понятно ли, где он будет использоваться;
* есть ли корректное название;
* не дублирует ли он уже существующий материал;
* можно ли использовать его в production без доработок.

Если материал не проходит проверку, его нельзя добавлять в активную библиотеку.

## 12. What Not To Add

Запрещено добавлять в бренд-материалы:

* случайные картинки “на всякий случай”;
* стоки без смысла;
* роботов;
* AI-сферы;
* мультяшные иллюстрации;
* неон;
* glassmorphism;
* generic office photos;
* слишком глянцевые изображения;
* файлы без понятного названия;
* дубли;
* устаревшие версии без пометки archive;
* изображения, которые не поддерживают HR Editorial Workspace;
* референсы без пояснения, что именно в них полезно;
* промпты без ограничений и запретов.

Библиотека бренд-материалов не должна превращаться в склад “может пригодиться”. Она должна быть рабочим инструментом.

## 13. Archive Rules

Если материал больше не используется, не удалять сразу.

Создавать внутри нужной папки подпапку:

`archive`

Туда переносить старые версии.

Правила архива:

* не хранить в активных папках несколько непонятных версий одного и того же файла;
* архивные файлы должны сохранять понятные названия;
* если причина архивации важна, рядом можно добавить короткую заметку;
* не использовать архивные материалы в production без повторной проверки;
* устаревшие версии логотипов, фото и mockups должны уходить в archive, а не оставаться рядом с актуальными файлами.

Пример:

```text
brand-materials/
└── logos/
    ├── hrneiroway-logo-primary.svg
    └── archive/
        └── hrneiroway-logo-primary-v01.svg
```

## 14. Final Rule

В бренд-материалы попадает только то, что помогает сделать сайт сильнее.

Если материал просто “красивый”, но не усиливает HRNEIROWAY — его не использовать.

Финальное правило:

каждый бренд-материал должен поддерживать хотя бы одну из задач:

* объяснить услугу;
* доказать экспертность;
* усилить доверие;
* поддержать HR Editorial Workspace;
* улучшить навигацию;
* сделать сайт визуально цельнее;
* помочь пользователю разобраться в сложной информации.

После этого документа проектную документацию не расширять без отдельной причины. Следующий этап — использовать созданную базу для подготовки материалов, структуры проекта и будущей разработки.
